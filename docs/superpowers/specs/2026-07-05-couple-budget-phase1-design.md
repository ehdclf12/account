# 부부 자산관리 앱 — 1단계 설계서 (가계부 코어)

작성일: 2026-07-05

## 0. 배경 & 전체 그림

부부 두 사람이 각자 폰에서 입력하고 서로의 내역을 함께 보는 **자산관리 앱**을 만든다.
형태는 **PWA**(홈 화면에 설치되는 웹앱)로, 앱스토어 등록 없이 두 사람이 바로 설치해 쓴다.

전체는 3개 시스템을 단계적으로 쌓는다. 본 설계서는 **1단계(가계부 코어)** 만 다룬다.

- **1단계 (본 문서):** 가계부 코어 — 지출/수입 입력, 카테고리, 누가 썼는지, 이번 달 요약
- 2단계: 예산 & 고정비 세부관리 & 공동/개인 정산
- 3단계: 투자자산(주식·코인 수량 입력 → 시세 자동 반영)
- 4단계: 부동산(수기) + 순자산 대시보드
- 5단계: 분석(그래프·검색·필터)

UI 전반은 **토스(Toss) 스타일**을 지향한다: 흰 배경, 큰 숫자, 둥근 카드, 넉넉한 여백, 파란 포인트색, 하단 탭, 화면당 정보 최소.

## 1. 1단계 범위

**포함**
- 공용 PIN 4자리 잠금 → 남편/아내 선택 (로그인/회원가입 없음)
- 지출/수입 입력 (금액, 날짜, 카테고리, 결제수단, 메모, 누가)
- 카테고리 관리 (기본 제공 + 추가/수정/삭제)
- 결제수단 관리 (기본 제공 + 추가/수정/삭제)
- 이번 달 요약(수입/지출/남은 돈) + 내역 목록(날짜별 그룹핑)
- 내역 수정/삭제
- 두 폰 간 실시간 공유

**미포함 (다음 단계)**
- 예산 설정·예산 대비 지출, 고정비 자동등록/세부관리 → 2단계 (1단계는 `is_fixed` 플래그 토대만)
- 공동/개인 정산 → 2단계
- 투자자산·부동산·순자산 → 3~4단계
- 그래프·통계 → 5단계

## 2. 인증 & 접근 방식 (초간단)

두 사람만 쓰므로 정식 회원가입/로그인을 두지 않는다.

**사용자 경험**
```
앱 첫 실행
 → 공용 PIN 4자리 입력 (두 사람 동일, 처음 한 번 뒤 폰이 기억)
 → "나는 누구?"  [ 남편 ]  [ 아내 ]  (한 번 선택 후 폰에 저장)
 → 바로 사용
```

**내부 동작 (보안)**
- 앱은 미리 만들어 둔 **하나의 공유 Supabase 계정**으로 자동 로그인한다. 이로써 DB 레벨(RLS)에서 실제 접근 제어가 걸린다.
- 공용 PIN은 클라이언트 측 잠금(주소를 아는 타인의 무단 접근 방지)이며, 폰의 로컬 저장소에 검증 상태를 기억한다.
- 선택한 남편/아내 정체성은 폰 로컬에 저장되어, 이후 입력의 `who` 값으로 자동 사용된다.

**정직한 한계**
- PIN은 가벼운 게이트이며 강력한 다중 사용자 인증은 아니다. 부부 전용 사설 앱 수준에서는 충분하다.
- "누가 썼는지"는 폰에서 고른 정체성 기반(신뢰 모델). 부부 간에는 문제없다.

## 3. 데이터 모델 (Supabase / Postgres)

두 사람만 쓰므로 별도 `households` 다중 테넌트는 두지 않고, **단일 공유 데이터셋**으로 단순화한다.

**profiles — 고정 2명**
| 컬럼 | 설명 |
|---|---|
| id | 고유 id |
| role | husband / wife |
| display_name | "남편" / "아내" (수정 가능) |

**categories**
| 컬럼 | 설명 |
|---|---|
| id | 고유 id |
| name | 식비, 교통 … |
| icon | 이모지/아이콘 키 |
| type | expense / income |
| is_fixed | 고정비 성격 여부 (2단계 토대) |
| sort_order | 정렬 순서 |

**payment_methods**
| 컬럼 | 설명 |
|---|---|
| id | 고유 id |
| name | 신용카드, 체크카드, 현금 … |
| icon | 아이콘 키 |

**transactions — 핵심**
| 컬럼 | 설명 |
|---|---|
| id | 고유 id |
| who | husband / wife (누가 썼는지) |
| type | expense / income |
| amount | 금액(원, 정수) |
| category_id | → categories |
| payment_method_id | → payment_methods |
| date | 사용 날짜 |
| memo | 메모 |
| created_at | 입력 시각 |

**설계 포인트**
- 모든 데이터는 공유 계정 소유 → 두 사람이 동일 내역을 실시간 열람/편집.
- `transactions.who` 로 "누가" 구분 → 2단계 정산의 기반.
- `categories.is_fixed` 로 고정비 표시 → 2단계 고정비 세부관리로 확장.
- **RLS**: 공유 Supabase 계정 소유 행만 접근 가능하도록 정책 설정.

**기본 시드 데이터**
- 카테고리(지출): 🍚식비, 🚕교통, 🛒생활, 🏠주거, 🎬여가, 💊의료, 💳고정비(is_fixed) 등
- 카테고리(수입): 💰월급, 💵기타수입
- 결제수단: 신용카드, 체크카드, 현금, 계좌이체
- profiles: 남편, 아내 2행

## 4. 화면 구성 (토스풍)

하단 탭 3개(홈 / 내역 / 설정) + 2단계에서 📊분석 추가. 입력은 + 플로팅 버튼 → 하단 시트.

**홈** — 이번 달 남은 돈(큰 숫자) · 수입/지출 카드 2개 · 최근 내역 · + 버튼
**입력 시트** — 큰 금액 입력 · 지출/수입 토글 · 카테고리 아이콘 칩 · 날짜/결제수단/메모/누가 · 저장 버튼
**내역** — 월 이동(‹ ›) · 상단 월 합계 · 날짜별 그룹핑 목록 · 지출 검정/수입 파랑
**설정** — 프로필(남/녀 이름), 카테고리 관리, 결제수단 관리, PIN 변경

(상세 아스키 목업은 브레인스토밍 대화 기록 참조.)

## 5. 기술 스택 & 배포

| 영역 | 선택 |
|---|---|
| 화면 | React + Vite + TypeScript |
| PWA | vite-plugin-pwa (홈 화면 설치, 전체화면) |
| 스타일 | Tailwind CSS |
| 데이터 | TanStack Query + Supabase JS (실시간 동기화) |
| 백엔드/DB | Supabase (Postgres + Auth(공유계정) + Realtime + RLS) |
| 배포 | Vercel(프론트) + Supabase(관리형), 둘 다 무료 등급 |

- 3단계 시세 프록시용 Supabase Edge Function 자리만 비워둔다(1단계 불필요).
- 배포 주소(예: `our-budget.vercel.app`)로 접속 후 두 폰에서 "홈 화면에 추가".

## 6. 개발 순서 (검증 포함)

1. 프로젝트 셋업(Vite+React+Tailwind+PWA) → 검증: 폰 홈화면 설치·실행
2. Supabase 연결 + 테이블·시드·RLS → 검증: 데이터 격리 확인
3. 공유계정 자동로그인 + PIN 게이트 + 남편/아내 선택 → 검증: PIN 통과·정체성 저장
4. 카테고리·결제수단 관리 화면 → 검증: 추가/수정/삭제 반영
5. 입력 시트(지출/수입) → 검증: 저장 시 DB·화면 즉시 반영
6. 홈 요약 + 내역 목록 + 수정/삭제 → 검증: 두 폰 실시간 동일 표시
7. 토스풍 UI 다듬기 + PWA 최종 점검 → 검증: 실제 폰 사용감

## 7. 성공 기준 (1단계 완성 정의)

- [ ] 두 사람이 각자 폰(홈 화면 앱)에서 PIN → 남편/아내 선택으로 진입한다
- [ ] 한 사람이 입력한 지출이 다른 사람 화면에도 나타난다(누가 포함)
- [ ] 홈에서 이번 달 수입·지출·남은 돈이 정확히 계산된다
- [ ] 카테고리·결제수단을 추가/수정/삭제할 수 있다
- [ ] 내역을 수정/삭제할 수 있다
- [ ] RLS로 공유 데이터 외 접근이 차단된다

## 8. 테스트

- 폰 2대(남편/아내)로 실제 공유 시나리오를 직접 검증.
- 월 합계·잔액 계산과 RLS 격리는 자동 테스트로도 확인.
