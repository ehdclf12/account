# 시간관리(블럭 타이머) — 설계

2026-07-19

## 목표

활동 "블럭"을 만들어두고 탭하면 그 활동의 타이머가 도는 시간 추적 기능.
사용자 제공 레퍼런스(블럭 생성 모달 + 균등 격자 + 점선 추가 카드) 방향을 따른다.

**기록은 개인별**이다. 두 사람이 같은 앱을 쓰지만 서로의 시간관리 기록은 보지 않는다.

## 범위

이 문서는 **1단계**만 다룬다.

- 1단계(이 문서): 블럭 CRUD, 타이머, 세션 기록, 오늘/이번 주 누적
- 2단계(별도): `/calendar` 하단 `캘린더 / [내 이름]` 모드 전환 + 월간 히트맵

1단계가 데이터를 만들어야 2단계 히트맵에 표시할 것이 생긴다. 순서를 뒤집으면 빈 화면이다.

## 데이터

```sql
time_blocks
  id, who, name(1~16자), emoji, color, sort_order, archived, created_at

time_sessions
  id, block_id → time_blocks(cascade), who, started_at, ended_at(null=진행중), created_at
```

### 진행 중인 타이머의 표현

**`ended_at`이 null인 세션 한 행**으로 표현한다. 로컬 상태로 들고 있으면 새로고침에 날아가고
기기를 바꾸면 이어지지 않는다. DB에 두면 앱을 껐다 켜도, 다른 기기에서도 이어진다.

경과 시간은 항상 `now - started_at`으로 계산한다. 그래서 **백그라운드로 가도 시간이 정확하다.**
(PWA 제약은 "끝났을 때 알림"에만 해당하며, 이 기능은 알림을 쓰지 않는다)

### 동시에 하나만

사람당 진행 중 세션은 하나로 **DB에서 강제**한다.

```sql
create unique index time_sessions_one_running on time_sessions (who) where ended_at is null;
```

앱에서도 막지만, 두 기기에서 동시에 시작하는 경합은 앱 로직으로 못 막는다.
부분 유니크 인덱스가 최후의 방어선이다.

### 접근 범위

RLS는 기존과 같이 `is_household_member()` 부부 스코프. 개인 분리는 **앱에서 `who = 나`로 조회**한다.
DB 레벨로 완전히 나누려면 정책에 이메일↔who 매핑이 필요한데, 둘만 쓰는 앱에서는 과하다고 판단했다.
(상대의 기록을 기술적으로 못 보게 해야 한다면 정책을 추가해야 한다 — 현재는 "앱이 안 보여줄 뿐")

## 화면

### `/time` — 블럭 격자

- 아카이빙 바둑판과 같은 `grid-cols-2` 균등 카드
- 카드: 이모지 + 이름 + 오늘 누적 시간, 색상 스트립
- 점선 테두리 `+` 카드로 블럭 추가
- **진행 중이면 상단 고정 배너** — 블럭명, 1초마다 흐르는 경과 시간, 정지 버튼
- 진행 중인 블럭 카드는 색으로 강조
- 상단에 오늘 / 이번 주 합계

### 블럭 생성·편집 시트

- 이모지, 작업명(16자 + `5/16` 카운터), 색상(`ARCHIVE_COLORS` 재사용)
- 편집 시 삭제 버튼(`confirm` 게이트) — 세션도 cascade 삭제됨을 문구로 알림

레퍼런스의 일러스트 아이콘 대신 **이모지**를 쓴다. 아이콘 세트를 만들거나 사올 필요가 없고
자유도가 높다.

### 홈 메뉴

`HomeMenuScreen`에 네 번째 항목 `시간관리` 추가 → `/time`

## 동작 규칙

- 블럭 탭 → 그 블럭 타이머 시작
- 다른 블럭 탭 → 이전 세션 종료 후 새 세션 시작(전환)
- 진행 중인 블럭 다시 탭 → 정지
- 누적 표시는 **오늘 / 이번 주**(월요일 시작)
- **자정 넘긴 세션은 시작일에 몰아서 집계.** 쪼개는 편이 정확하지만 예측 가능성이 떨어진다.
  타이머를 자정 너머로 돌리는 일은 드물다고 보고 단순한 쪽을 택한다.
- 진행 중인 세션도 오늘 누적에 실시간 반영

## 주 시작 요일

**월요일 시작으로 통일한다.** 레퍼런스 스크린샷은 일요일 시작이지만, 기존
`CalendarScreen`이 `WEEK = ['월'…'일']`로 월요일 시작이다. 2단계에서 같은 화면을 토글로
오가는데 요일 순서가 바뀌면 혼란스럽다. 앱 내 일관성을 우선한다.

## 재사용

새로 만드는 것이 생각보다 적다. 아카이빙에서 검증된 균등 그리드 레이아웃, `ARCHIVE_COLORS`,
시트 모달 패턴, react-query 훅 구조, 그리고 최근 추가한 토스트·전역 오류 처리를 그대로 쓴다.

## 테스트 (순수 함수 TDD)

`src/lib/time.test.ts`
- `elapsedSeconds`: 종료된 세션 / 진행 중(now 기준) / started_at이 미래인 이상값 → 0
- `formatDuration`: 초 단위 / 분 / 시간+분 / 0 / 큰 값
- `totalsByBlock`: 블럭별 합산, 진행 중 세션 포함, 다른 사람 세션 제외
- `sumInRange`: 오늘 / 이번 주(월요일 시작) 경계, 자정 넘긴 세션이 시작일에 잡히는지
- 주 경계: 월요일 00:00 / 일요일 23:59

기존 140개 테스트는 그대로 통과해야 한다.

## 실패 처리

블럭·세션 변경은 mutation이므로 실패 시 전역 토스트가 이미 사유를 알린다.
진행 중 세션 시작이 유니크 인덱스에 걸리면(다른 기기에서 이미 시작) `23505`가 오고
"이미 등록된 항목이에요."로 안내된다 — 이 경우만 문구를 따로 다듬는다.
